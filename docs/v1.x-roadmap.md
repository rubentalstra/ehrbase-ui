# v1.x Roadmap — features deliberately deferred past v1.0

> Single place to track every UI / capability that was considered for v1.0, decided important enough to remember, but deferred to v1.x for a specific reason. Each item lists _what_ it would deliver, _why_ it isn't in v1.0, and _what would have to land first_ for it to ship.

This document is **not a wish-list**. Items here are decisions, ratified at the milestone-planning level (see `docs/IMPLEMENTATION_CHECKLIST.md` for v1.0 scope, and `docs/architecture.md` §1 for the "Explicitly NOT in v1.0" header).

Updated cadence: appended to whenever a milestone PR identifies a feature that doesn't fit v1.0 and we want to remember it.

---

## Scheduling / Appointments

**What it would deliver:** clinic calendar view (per clinician / per room / per specialty), appointment slots, waitlist management, recurring-appointment rules, rescheduling UI, no-show handling, appointment requests from patients (via the patient portal — also v1.x).

**Why it's deferred from v1.0:** scheduling is its own product. HIX, Epic, and Cerner each spend a significant share of their UX budget on it; underestimating that surface is a common trap for early EPDs. Also, openEHR has no first-class scheduling model — it would be either ADMIN_ENTRY-based (loose) or a parallel non-openEHR module. Either way, it deserves its own ADR + architecture chapter when the time comes.

**openEHR alignment:** appointments are NOT a natural fit for OBSERVATION / EVALUATION / INSTRUCTION / ACTION. The closest openEHR construct is ADMIN_ENTRY (administrative data) plus the **PROC component** (`WORK_PLAN` / `TASK_PLAN` / `PLAN_ITEM`) for repeating tasks. A v1.x scheduling module would likely combine: ADMIN_ENTRY for the appointment record, PROC for the recurring/series logic, and a non-openEHR Postgres table for the per-clinician calendar grid (since the grid is a view, not clinical data).

**What blocks it from v1.0:** scope. Building scheduling well takes ~2–3 dedicated milestones (calendar UI, slot/availability logic, rescheduling + waitlist, recurring rules). Adding it to v1.0 would either delay every other clinical surface or ship a thin/broken scheduling.

**Relevant ADRs (when implemented):** new ADR `0026-scheduling-data-model` (openEHR ADMIN_ENTRY + PROC vs custom Postgres). Possibly `0027-calendar-ui-library` (FullCalendar vs react-big-calendar vs custom).

## Embedded DICOM viewer

**What it would deliver:** in-app DICOM image viewer (single-frame + multi-frame, MPR, basic windowing/level, measurement tools). User stays in the EPD for radiology review.

**Why it's deferred from v1.0:** DICOM viewing is a whole product area (Cornerstone.js, OHIF, Weasis — each is its own community of practice). Hospitals already run a PACS with its own viewer; v1.0 ships a **DICOM-listing + external-viewer launcher** instead (per decision #6 / ADR-0020): we display DICOM study metadata + a "Open in PACS viewer" link that hands off to whatever the deployment has configured.

**openEHR alignment:** DICOM references can sit inside a COMPOSITION via `DV_MULTIMEDIA` (`mimetype: application/dicom`) or as a `URI` link in an OBSERVATION (`openEHR-EHR-OBSERVATION.imaging_examination_result.v1`).

**What blocks it from v1.0:** scope + decision to favour hospitals' existing PACS investment in v1.0.

**Relevant ADRs (when implemented):** new ADR `0028-dicom-viewer-integration` (Cornerstone.js vs OHIF; in-process vs iframe; toolset).

## Real-time updates (WebSocket / SSE)

**What it would deliver:** new lab results, new orders, new messages, allergy/critical-result alerts arrive at the open patient view without manual refresh. The flowsheet ticks. Inbox badges increment live.

**Why it's deferred from v1.0:** persistent server-push channels are their own category of DPIA risk (channel can leak presence / activity metadata). v1.0 uses TanStack Query polling at sensible intervals (5–30 s for active views, longer for background). Sufficient for v1.0 scale.

**openEHR alignment:** openEHR has no real-time push spec; this is purely an implementation concern.

**What blocks it from v1.0:** a dedicated DPIA addendum covering the persistent channel + ops work (Redis pubsub / WebSocket gateway sizing).

**Relevant ADRs (when implemented):** new ADR `0029-real-time-update-channel` (WS vs SSE; per-user channel topology; back-pressure).

## AI / LLM clinical decision support

**What it would deliver:** beyond v1.0's rule-based GDL2-aligned evaluator — natural-language summarisation of the chart, differential-diagnosis suggestions, drug-interaction reasoning beyond table lookups, retrieval-augmented Q&A over the patient's own record.

**Why it's deferred from v1.0:** GDPR Art. 22 (automated decision-making) considerations are non-trivial; needs its own DPIA. Patient consent boundaries are unclear in the EU regulatory landscape (EHDS Art. 5–6 evolving). Liability if a recommendation is wrong is real. v1.0 ships deterministic rule-based CDS only, fully auditable. AI CDS is a v1.x or v2 capability that ships with its own legal envelope.

**openEHR alignment:** AI outputs could be persisted as EVALUATION (with `provider` = AI agent identifier) but the openEHR community is still debating provenance / explainability fields.

**What blocks it from v1.0:** legal envelope (separate DPIA), liability framework, vendor evaluation, deterministic-rules baseline must come first.

**Relevant ADRs (when implemented):** new ADRs `0030-ai-cds-provenance` and `0031-ai-cds-consent-model`.

## Native iOS / Android apps

**What it would deliver:** native bedside clinical apps. Push notifications. Offline drafts. Biometric login.

**Why it's deferred from v1.0:** the web app is fully usable on tablet at the bedside (§12 target-size compliance enforces 24-px minimum). Native is a separate product; it duplicates the front-end surface in a different toolchain.

**What blocks it from v1.0:** the web tablet experience must be production-grade first (§12 manual NVDA + VoiceOver passes, target-size axe gates).

## Offline / PWA mode

**What it would deliver:** clinician can keep working when the network drops; drafts queue locally; sync on reconnect.

**Why it's deferred from v1.0:** hospital deployments are connected by design. Home-visit nursing is a separate product configuration with different consent + security posture.

## Real GDL2 engine integration

**What it would deliver:** replace v1.0's native rule evaluator with a full openEHR GDL2 execution engine. Rules become portable to any GDL2-aware system.

**Why it's deferred from v1.0:** no production-grade open-source GDL2 engine exists in the ecosystem yet. v1.0's native evaluator uses a GDL2-aligned rule format (archetype binding + pre-conditions + actions) — when an engine matures, the rules port mechanically.

**Relevant ADRs (when implemented):** update ADR-0021 (CDS scope) with a status transition; possibly new ADR for the chosen engine.

## External Patient Master Index (HL7 v2 ADT integration)

**What it would deliver:** for hospital deployments that already run a PMI fed by HL7 v2 ADT feeds, the M7 openEHR-spec demographic service becomes a _consumer_ of the PMI rather than the source-of-truth. ADT events update demographics in near-real-time.

**Why it's deferred from v1.0:** v1.0 ships the M7 demographic service as the spec-pure greenfield option (decision #9). Production hospitals will need the PMI adapter; the adapter contract is openEHR-PARTY-shaped so the swap is mechanical.

**openEHR alignment:** the PARTY model accepts any external identifier via `external_ref` — ADT feeds map cleanly to PERSON / PARTY_IDENTITY / CONTACT / ADDRESS.

**Relevant ADRs (when implemented):** new ADR `0032-pmi-adt-integration` (FHIR Mirth integration vs direct HL7 v2 parsing; conflict-resolution rules between the PMI and our store).

## EHDS cross-border features (MyHealth@EU)

**What it would deliver:** patient summary, ePrescription, eDispensation exchangeable across EU member states per the EHDS Regulation (EU 2025/327) timeline: secure-processing-environment requirements 26 Mar 2027; patient summaries / ePrescriptions / eDispensations 26 Mar 2029; medical images / lab results / discharge reports 26 Mar 2031.

**Why it's deferred from v1.0:** the EHDS timeline puts this beyond v1.0's tag-by-2026 plan. We ship the _data layer_ compatible (openEHR + AQL + CANONICAL JSON export) so the cross-border surface plugs in later.

**openEHR alignment:** EHDS is increasingly openEHR-aware; the European Patient Summary maps to openEHR archetypes.

**Relevant ADRs (when implemented):** new ADR `0033-ehds-myhealth-eu-bridge`.

---

## Patient portal

**What it would deliver:** patient-facing UI beyond `/me/access-log` — view own record, secure messaging with care team, appointment requests (depends on Scheduling), document download (Art. 20 portability), consent management UI.

**Why it's deferred from v1.0:** v1.0 is the clinical-facing workspace. The patient portal is a separate front-end product (different user, different auth, different threat model). v1.0 ships `/me/access-log` (Art. 15) as the patient-facing minimum.

**Relevant ADRs (when implemented):** new ADR `0034-patient-portal-architecture` (same codebase + role-based UI vs separate front-end).

## Audit & observability hardening (beyond the v1.0 IHE ATNA trail)

**What v1.0 already ships (NOT deferred):** the **IHE ATNA access-audit trail** — every PHI read /
write / query emits an IHE-ATNA-conformant DICOM AuditMessage from the BFF into a queryable Postgres
`audit` schema, with an optional syslog forwarder; plus the openEHR-native `CONTRIBUTION` /
`AUDIT_DETAILS` / `ATTESTATION` write lineage. Built in **M9** ([ADR-0041](adr/0041-audit-access-governance.md)),
consumed by the M22 audit-review dashboard + Article-15 patient access-log.

**What it would deliver (deferred enhancements on top of that trail):**

- **Tamper-evidence** — a hash chain over the `audit` table so the access trail is provably
  unaltered.
- **Retention + purge** — configurable national-law retention windows with a tagged purge job
  (reconciling GDPR minimisation with clinical-record-retention law).
- **Cold-store WORM tier** — write-once archival of aged audit + clinical data (e.g. S3 Object Lock).
- **Full observability** — the OpenTelemetry SDK + Tempo / Loki / Prometheus / Grafana stack with
  the four-layer PHI-redaction pipeline (v1.0 keeps only `/api/health` + `/api/ready` probes and
  plain-stdout Pino logging).

**Why deferred from v1.0:** none of these block a correct, standards-conformant access trail — they
harden it. They matter before a production deployment touches real patient data, but not for the
clinical-build milestones. (This is the residue of the removed governance layer — see `CLAUDE.md` →
"Deferred (post-core)". The core-refocus removed the _bespoke NEN-7513 hash-chain subsystem_; the
standards-based ATNA trail that replaces it ships in v1.0, M9.)

**openEHR / standards alignment:** the access trail is IHE ATNA (DICOM Audit Message Format);
NEN-7513 compliance is satisfied by that trail ([ADR-0041](adr/0041-audit-access-governance.md)).
The hardening items are deployment/ops concerns, not openEHR-modelled.
